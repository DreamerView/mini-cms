# Mini Headless CMS

Лёгкая headless CMS, которая запускается из одного PHP-файла и подходит для обычного PHP-хостинга без Composer, Node.js, Docker и обязательного VPS.

Главная идея проекта — быстро подключить управляемый контент, формы и JSON API к сайту на HTML, Vue, Nuxt, React, Next.js или любом другом frontend-стеке, не устанавливая тяжёлую CMS.

> Проект находится в активной разработке. Перед использованием на рабочем сайте обязательно сделайте резервную копию и проверьте диагностику установки.

## Для чего подходит

Mini Headless CMS удобно использовать для:

- корпоративных сайтов и лендингов;
- каталогов товаров и услуг;
- блогов и новостных разделов;
- страниц с мультиязычным контентом;
- FAQ, контактов, преимуществ и других повторяемых блоков;
- приёма заявок с сайта;
- небольших клиентских проектов на обычном PHP-хостинге;
- frontend-проектов, которым нужен простой JSON API.

CMS не пытается заменить CRM, интернет-магазин, LMS или крупную корпоративную платформу. Она решает одну основную задачу: хранить и отдавать контент, файлы и заявки через понятную административную панель и API.

## Основные возможности

### Простая установка

- один основной PHP-файл;
- первый запуск через мастер настройки;
- выбор SQLite или MySQL;
- автоматическое создание таблиц и обновление структуры базы;
- не нужен Composer;
- не нужен отдельный backend-сервер;
- не нужен cron для базового обслуживания.

### Проекты

В одной установке можно создать несколько независимых проектов.

Для каждого проекта отдельно хранятся:

- коллекции;
- разделы контента;
- записи;
- формы и заявки;
- файлы;
- CORS-настройки;
- API-ключи;
- Telegram-интеграция;
- права пользователей.

### Коллекции

Коллекция — самостоятельный источник данных.

Поддерживаются два типа:

- `Single` — одна запись, например главная страница или контакты;
- `Multiple` — несколько записей, например новости, товары или сотрудники.

Доступные возможности:

- создание и редактирование схемы;
- drag-and-drop сортировка;
- пресеты коллекций;
- клонирование коллекции;
- импорт и экспорт схемы;
- публичный и приватный доступ;
- именованные API-ключи;
- поиск, фильтрация, сортировка и пагинация в API.

### Разделы контента

Разделы работают как подборки или ярлыки.

Одна коллекция может использоваться сразу в нескольких разделах. При удалении раздела сама коллекция, её поля и записи не удаляются.

Например, коллекцию `products` можно одновременно подключить к разделам:

- Главная страница;
- Каталог;
- Популярные товары.

### Поля коллекций

Поддерживаются типы:

- `text`;
- `textarea`;
- `html`;
- `email`;
- `tel`;
- `number`;
- `integer`;
- `date`;
- `datetime`;
- `bool`;
- `url`;
- `image`;
- `file`;
- `json`;
- `relation`.

Для полей доступны:

- обязательность;
- минимальная и максимальная длина;
- минимальное и максимальное значение;
- регулярное выражение;
- допустимые значения;
- значение по умолчанию;
- уникальность;
- связь с другой коллекцией;
- одиночная или множественная relation-связь.

### Записи

- статусы `draft` и `published`;
- автоматическое создание slug;
- автосохранение черновика;
- восстановление несохранённого черновика;
- история изменений;
- восстановление предыдущей версии;
- JSON-предпросмотр;
- HTML-предпросмотр;
- раскрытие relation-полей через `populate`;
- быстрый переход к API endpoint.

Публичный API отдаёт только опубликованные записи.

### Интернационализация

Интерфейс административной панели доступен на:

- русском;
- казахском;
- английском.

Для контента можно включить отдельные языки, включая:

`ru`, `kk`, `en`, `tr`, `de`, `fr`, `es`, `it`, `pt`, `zh`, `ja`, `ko`, `ar`, `hi`, `uz`.

Язык ответа API выбирается параметром `lang`.

### Файлы

- загрузка изображений и документов;
- максимальный размер по умолчанию — 10 МБ;
- белый список расширений;
- регистрация файлов в базе;
- поиск используемых и неиспользуемых файлов;
- корзина;
- восстановление;
- окончательное удаление;
- глобальные orphan-файлы;
- привязка файлов к проектам.

### Формы и заявки

CMS может принимать формы с внешнего сайта через публичный endpoint.

Поддерживаются форматы:

- `application/x-www-form-urlencoded`;
- `multipart/form-data` без файлов;
- `application/json`.

Типы полей формы:

- текст;
- большой текст;
- email;
- телефон;
- число;
- целое число;
- boolean;
- дата;
- дата и время;
- URL;
- JSON.

Сервер проверяет:

- обязательные поля;
- тип значения;
- неизвестные поля;
- максимальную длину;
- корректность email, телефона, URL, даты и JSON;
- размер всего payload.

Дополнительно доступны:

- защита honeypot;
- rate limit;
- статусы `new`, `read`, `spam`;
- массовое изменение статуса;
- массовое удаление;
- поиск и фильтры;
- пагинация;
- экспорт в CSV и JSON;
- автоматическое удаление прочитанных заявок и спама через 30, 90, 180 или 365 дней;
- email-уведомления через PHP `mail()`;
- webhook с HMAC-подписью;
- Telegram-уведомления.

### Telegram-бот

Telegram-интеграция настраивается отдельно для каждого проекта.

Текущий сценарий подключения:

1. Создайте бота через BotFather.
2. Вставьте токен в конфигуратор CMS.
3. Нажмите «Проверить бота».
4. Нажмите «Открыть бота и ждать».
5. Отправьте боту любое сообщение в личном чате.
6. Для группы отправьте команду `/chatid`.
7. Бот ответит Chat ID, а CMS автоматически подставит его в поле.
8. Отправьте тестовое сообщение и включите уведомления.

При новой заявке бот отправляет:

- название проекта;
- название формы;
- номер заявки;
- дату и время;
- до 20 заполненных полей;
- URL страницы отправки, если он доступен.

Особенности:

- ошибка Telegram не мешает сохранить заявку;
- ошибки доставки записываются в журнал действий;
- токен бота хранится в зашифрованном виде;
- для получения Chat ID модальное окно CMS должно оставаться открытым;
- `getUpdates` не работает, если у бота уже настроен webhook.

### API

API поддерживает:

- индекс доступных маршрутов;
- список записей;
- одну запись по slug;
- раздел с несколькими коллекциями;
- выбор языка;
- relation populate;
- поиск;
- фильтрацию;
- сортировку;
- пагинацию;
- выбор возвращаемых полей;
- ETag и HTTP-кэширование;
- публичный и приватный режим;
- CORS.

### API-ключи

Для коллекций, разделов и форм можно включить приватный доступ.

Именованные ключи поддерживают:

- отдельное название;
- срок действия;
- дату последнего использования;
- отзыв ключа;
- отображение полного значения только один раз;
- хранение только хеша ключа в базе.

Ключ можно передавать двумя способами:

```http
X-API-Key: cms_xxxxxxxxxxxxxxxxx
```

или:

```http
Authorization: Bearer cms_xxxxxxxxxxxxxxxxx
```

### Пользователи и роли

Поддерживаются роли:

- `Administrator` — полный доступ;
- `Developer` — схема, связи, API, настройки и диагностика;
- `Editor` — записи, формы, заявки и файлы;
- `Viewer` — просмотр контента.

Права могут назначаться отдельно внутри каждого проекта.

### Резервные копии

Резервная копия проекта содержит:

- структуру коллекций;
- поля;
- записи;
- разделы контента;
- формы;
- поля форм;
- заявки;
- файлы проекта.

Приватные API-ключи в архив не включаются.

Восстановление выполняется как новый проект, поэтому существующий проект не перезаписывается.

Для резервных копий требуется расширение `ZipArchive`.

### Система и контроль

- журнал действий;
- IP и пользователь для изменений;
- диагностика PHP и расширений;
- проверка прав записи;
- проверка HTTPS;
- проверка защиты каталога `storage`;
- Debug Mode;
- автоматическая очистка старых черновиков и потерянных версий;
- очистка заявок по сроку хранения;
- светлая и тёмная тема;
- адаптивная административная панель.

## Системные требования

Обязательно:

- PHP `8.1` или новее;
- PDO;
- `mbstring`;
- `json`;
- `openssl`;
- `fileinfo`;
- `pdo_sqlite` или `pdo_mysql`;
- права на запись в каталоги `storage` и `uploads`.

Рекомендуется:

- HTTPS;
- `ZipArchive` для резервных копий;
- cURL для Telegram и webhook-запросов;
- `upload_max_filesize` не меньше 10 МБ;
- `post_max_size` не меньше 10 МБ.

Если cURL недоступен, CMS пытается использовать `file_get_contents` с HTTP stream context.

Административная панель загружает Bootstrap 5.3 и Bootstrap Icons через CDN, поэтому для полного оформления нужен доступ к интернету.

## Установка

### 1. Загрузите файл

Переименуйте основной файл CMS в `index.php` и загрузите его в отдельную папку на хостинге.

Пример:

```text
/public_html/cms/index.php
```

### 2. Создайте каталоги

```text
/public_html/cms/storage
/public_html/cms/uploads
```

На Linux-хостинге обычно достаточно:

```bash
chmod 775 storage uploads
```

Владелец каталогов должен совпадать с пользователем, от которого работает PHP.

### 3. Откройте CMS в браузере

```text
https://example.com/cms/
```

При первом запуске мастер предложит:

1. выбрать SQLite или MySQL;
2. сохранить параметры подключения;
3. создать первого администратора;
4. открыть панель проекта.

### 4. Проверьте диагностику

После установки откройте:

```text
Настройки → Интеграции и контроль → Диагностика установки
```

Исправьте все пункты со статусом «Ошибка».

## SQLite

SQLite подходит для небольших и средних проектов, где важна максимально простая установка.

Файл базы создаётся автоматически:

```text
storage/cms.sqlite
```

Преимущества:

- не нужно заранее создавать базу;
- простой перенос проекта;
- удобная локальная разработка;
- минимум настроек.

Для сайта с высокой конкурирующей записью или большим количеством заявок лучше использовать MySQL.

## MySQL

Перед установкой создайте пустую базу данных и пользователя.

Пример параметров:

```text
Host: localhost
Database: mini_cms
User: mini_cms_user
Password: strong-password
```

CMS сама создаст необходимые таблицы.

Рекомендуется MySQL 8 или совместимая версия MariaDB с `utf8mb4`.

## Структура файлов

```text
cms/
├── index.php
├── storage/
│   ├── config.json
│   ├── cms.sqlite
│   ├── login_attempts.json
│   └── .htaccess
└── uploads/
```

Некоторые файлы появляются только после первого запуска.

### Важно для Nginx

`.htaccess` работает только в Apache. Для Nginx запретите прямой доступ к `storage` вручную:

```nginx
location ^~ /cms/storage/ {
    deny all;
    return 403;
}
```

Каталог `uploads` должен оставаться доступным, если frontend получает публичные файлы напрямую.

## Примеры API

Во всех примерах используется проект со slug `default`.

### Индекс API

```http
GET /cms/?api=index&project=default
```

### Список записей

```http
GET /cms/?api=entries&project=default&c=articles
```

### Одна запись

```http
GET /cms/?api=entry&project=default&c=articles&s=hello-world
```

### Раздел контента

```http
GET /cms/?api=group&project=default&g=home-page&lang=ru&populate=1
```

### Пагинация

```http
GET /cms/?api=entries&project=default&c=articles&page=1&limit=25
```

Максимальный `limit` — 100.

### Поиск

```http
GET /cms/?api=entries&project=default&c=articles&q=telegram
```

### Фильтр по полю

```http
GET /cms/?api=entries&project=default&c=articles&filter[category]=news
```

### Сортировка

```http
GET /cms/?api=entries&project=default&c=products&sort=-data.price
```

Знак `-` означает сортировку по убыванию.

### Выбор полей

```http
GET /cms/?api=entries&project=default&c=articles&fields=title,slug,data.image
```

### Раскрытие связей

```http
GET /cms/?api=entry&project=default&c=articles&s=hello-world&populate=1
```

### Язык контента

```http
GET /cms/?api=entries&project=default&c=articles&lang=kk
```

## Приватный API

Пример запроса с ключом:

```bash
curl "https://example.com/cms/?api=entries&project=default&c=private-content" \
  -H "X-API-Key: cms_your_key"
```

Через Bearer:

```bash
curl "https://example.com/cms/?api=entries&project=default&c=private-content" \
  -H "Authorization: Bearer cms_your_key"
```

API-ключ должен принадлежать конкретному ресурсу. Ключ одной коллекции не открывает другую приватную коллекцию.

## CORS

CORS настраивается отдельно для каждого проекта.

По умолчанию используется:

```text
*
```

Для ограничения доступа укажите один Origin на строку:

```text
https://site.example.com
https://admin.example.com
http://localhost:3000
```

Учитываются:

- протокол;
- домен;
- порт.

Пути и query-параметры в Origin не допускаются.

## Подключение HTML-формы

Предположим, создана форма со slug `contact` в проекте `default`.

```html
<form method="post" action="https://example.com/cms/?form=contact&project=default">
  <div class="mb-3">
    <label class="form-label" for="name">Имя</label>
    <input class="form-control" id="name" name="name" required>
  </div>

  <div class="mb-3">
    <label class="form-label" for="email">Email</label>
    <input class="form-control" id="email" name="email" type="email" required>
  </div>

  <div class="mb-3">
    <label class="form-label" for="message">Сообщение</label>
    <textarea class="form-control" id="message" name="message" required></textarea>
  </div>

  <div class="d-none" aria-hidden="true">
    <label for="website">Не заполняйте это поле</label>
```

## Отправка формы в JSON

```bash
curl -X POST "https://example.com/cms/?form=contact&project=default" \
  -H "Content-Type: application/json" \
  -d '{
    "name": "Темирхан",
    "email": "user@example.com",
    "message": "Хочу обсудить проект"
  }'
```

Успешный ответ:

```json
{
  "ok": true,
  "submission_id": 15,
  "message": "Спасибо! Форма успешно отправлена."
}
```

Ошибка валидации возвращает HTTP-код `422` и информацию о проблемном поле.

## Webhook формы

Для формы можно указать URL и секрет.

CMS отправляет JSON-событие:

```json
{
  "event": "form.submitted",
  "submission_id": 15,
  "status": "new",
  "project": {
    "id": 1,
    "name": "Основной сайт",
    "slug": "default"
  },
  "form": {
    "id": 3,
    "name": "Обратная связь",
    "slug": "contact"
  },
  "data": {
    "name": "Темирхан",
    "email": "user@example.com"
  },
  "received_at": "2026-06-22 12:00:00"
}
```

Заголовки:

```http
X-CMS-Event: form.submitted
X-CMS-Signature: sha256=...
```

Подпись рассчитывается как HMAC SHA-256 от исходного JSON с использованием секрета формы.

Пример проверки в PHP:

```php
<?php
$secret = 'your-webhook-secret';
$payload = file_get_contents('php://input');
$received = $_SERVER['HTTP_X_CMS_SIGNATURE'] ?? '';
$expected = 'sha256=' . hash_hmac('sha256', $payload, $secret);

if (!hash_equals($expected, $received)) {
    http_response_code(401);
    exit('Invalid signature');
}

$event = json_decode($payload, true, 32, JSON_THROW_ON_ERROR);
```

## Безопасность

В CMS используются:

- PDO prepared statements;
- `password_hash()` и `password_verify()`;
- CSRF-токены для административных действий;
- строгий режим PHP-сессий;
- `HttpOnly` cookie;
- `SameSite=Lax`;
- флаг `Secure` при HTTPS;
- ограничение попыток входа;
- проверка типов и размеров загружаемых файлов;
- whitelist расширений;
- серверная валидация форм;
- honeypot;
- rate limit;
- раздельные роли и права;
- хеширование API-ключей;
- журнал действий;
- защита `storage` через `.htaccess` в Apache.

### Рекомендации для production

- используйте HTTPS;
- не включайте Debug Mode на рабочем сайте;
- защитите каталог `storage` на уровне веб-сервера;
- регулярно скачивайте резервные копии;
- используйте MySQL для проектов с активной записью;
- создавайте отдельный API-ключ для каждого клиента;
- задавайте срок действия ключей;
- ограничивайте CORS конкретными доменами;
- не публикуйте Telegram-токен и `config.json`;
- проверьте права на `storage` и `uploads`.

## Debug Mode

Включается в настройках системы.

При включении PHP показывает подробные ошибки, предупреждения и stack trace стандартными средствами PHP.

Используйте режим только во время разработки. Сообщения PHP могут раскрыть:

- абсолютные пути к файлам;
- параметры сервера;
- структуру приложения;
- части конфигурации.

Синтаксическая ошибка в самом `index.php`, возникшая до запуска CMS, может потребовать включения `display_errors` через `php.ini`, `.user.ini` или панель хостинга.

## Обслуживание без cron

CMS периодически запускает обслуживание во время обычных запросов.

Очистка включает:

- старые автосохранённые черновики;
- версии, потерявшие запись или коллекцию;
- прочитанные заявки и спам старше срока хранения.

Также обслуживание можно запустить вручную в настройках.

---

## Ограничения текущей версии

- основной код находится в одном PHP-файле, поэтому расширение архитектуры требует аккуратной работы;
- endpoint форм пока не принимает файлы;
- Telegram-получение Chat ID требует открытой вкладки CMS;
- Telegram `getUpdates` несовместим с активным webhook того же бота;
- email-уведомления зависят от настройки PHP `mail()` на хостинге;
- резервные копии требуют `ZipArchive`;
- Bootstrap и Bootstrap Icons загружаются через CDN;
- проект пока развивается и требует дополнительного тестирования под высокой нагрузкой.

## Философия проекта

Mini Headless CMS создаётся как практичный инструмент для разработчиков и небольших команд, которым нужна простая CMS без лишней инфраструктуры.

Основные принципы:

- один файл проще развернуть;
- CMS должна решать CMS-задачи;
- frontend не должен зависеть от конкретного фреймворка;
- API должен оставаться понятным;
- обычный PHP-хостинг должен быть полноценной платформой;
- функции должны добавляться только тогда, когда они действительно упрощают работу.
