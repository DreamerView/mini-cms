# Mini Headless CMS

Минималистичная headless CMS на PHP для быстрого создания API-контента без тяжёлых админ-панелей. Проект рассчитан на простую установку, работу с SQLite или MySQL, управление коллекциями, записями, группами, файлами, мультиязычным контентом и связями между коллекциями.

CMS можно использовать как лёгкую админку для сайтов на обычном HTML, Vue, Nuxt, React, Next.js или любом другом фронтенде, который умеет получать данные через JSON API.

## Возможности

* Один PHP-файл без сложной сборки.
* Первый экран установки с выбором базы данных.
* Поддержка SQLite и MySQL.
* Хранение конфигурации в `storage/config.json`.
* Создание первого администратора после выбора базы.
* Проекты как отдельные workspace внутри одной базы данных.
* Группы коллекций для удобного API-запроса сразу нескольких коллекций.
* Коллекции двух типов:

  * `multiple` — много записей;
  * `single` — одна запись, удобно для главной страницы, настроек сайта, контактов.
* Поля разных типов:

  * `text`
  * `textarea`
  * `html`
  * `number`
  * `date`
  * `bool`
  * `url`
  * `image`
  * `file`
  * `json`
  * `relation`
* Записи со статусами:

  * `draft`
  * `published`
* Публичный JSON API для опубликованных записей.
* Приватные API-разделы для схемы, коллекций, групп и файлов.
* Связи между коллекциями через поле `relation`.
* Проверка, что связанная запись принадлежит выбранной целевой коллекции.
* Режим раскрытия связей через `populate=1`.
* Мультиязычный контент через `lang` или `locale`.
* Раскрытие связей внутри переводов при `populate=1`.
* Загрузка файлов и изображений.
* Очистка неиспользуемых файлов.
* Пользователи и роли.
* Защита форм через CSRF.
* Ограничение попыток входа.
* Светлая и тёмная тема.
* Интерфейс на русском, казахском и английском.
* Контентные языки: русский, казахский, английский, турецкий, немецкий, французский, испанский, итальянский, португальский, китайский, японский, корейский, арабский, хинди, узбекский.

## Быстрый старт

### 1. Скопируйте файл CMS

Например:

```text
/public_html/index.php
```

или:

```text
/var/www/site/index.php
```

### 2. Создайте папки

```bash
mkdir -p storage uploads
chmod -R 775 storage uploads
```

### 3. Запустите локально

```bash
php -S localhost:8000
```

Откройте:

```text
http://localhost:8000
```

### 4. Выберите базу данных

При первом запуске CMS покажет экран выбора базы:

* SQLite — простой вариант по умолчанию.
* MySQL — если нужна отдельная серверная база.

### 5. Создайте первого пользователя

После выбора базы CMS предложит создать первого администратора.

Пароль должен быть от 10 до 72 символов.

## Первый запуск

Первый запуск проходит в два шага.

### Шаг 1. Выбор базы данных

CMS создаёт файл:

```text
storage/config.json
```

Для SQLite конфиг будет примерно таким:

```json
{
  "db": {
    "driver": "sqlite",
    "sqlite_path": "/path/to/project/storage/cms.sqlite"
  },
  "created_at": "2026-06-18 10:00:00",
  "updated_at": "2026-06-18 10:00:00"
}
```

Для MySQL:

```json
{
  "db": {
    "driver": "mysql",
    "mysql": {
      "host": "localhost",
      "database": "cms",
      "user": "root",
      "password": "",
      "charset": "utf8mb4"
    }
  },
  "created_at": "2026-06-18 10:00:00",
  "updated_at": "2026-06-18 10:00:00"
}
```

Для MySQL базу данных нужно создать заранее.

### Шаг 2. Первый администратор

Если таблица пользователей пустая, CMS показывает форму создания первого пользователя.

Поля:

* логин;
* имя;
* пароль.

Первый пользователь получает роль `admin`.

## Проекты

Проект — это отдельный workspace внутри одной базы данных.

Пример:

* `default`
* `site-main`
* `client-a`
* `client-b`

У каждого проекта свои:

* коллекции;
* группы;
* записи;
* файлы.

Проекты полезны, если нужно держать несколько CMS-пространств в одной базе.

### Что можно делать с проектами

* создать проект;
* редактировать название, slug, описание и порядок;
* переключаться между проектами;
* удалить проект, если он не последний и не активный.

### Ограничения

Нельзя удалить:

* последний проект;
* активный проект.

Чтобы удалить активный проект, сначала переключитесь на другой.

---

## Группы

Группа — это набор коллекций, который можно получить одним API-запросом.

Пример:

```text
main
```

В группу `main` можно добавить:

* `pages`
* `services`
* `faq`
* `contacts`

После этого фронтенд может получить данные одним запросом:

```text
/?api=group&g=main
```

### Зачем нужны группы

Группы удобны, когда фронтенду нужно сразу несколько блоков контента.

Например, главная страница сайта может состоять из:

* hero;
* advantages;
* services;
* FAQ;
* footer contacts.

Без группы пришлось бы делать несколько запросов. С группой можно получить всё одним запросом.

### Управление группами

В админке можно:

* создать группу;
* редактировать группу;
* удалить группу;
* выбрать коллекции, входящие в группу;
* открыть API группы.

Удаление группы не удаляет коллекции и записи.

## Коллекции

Коллекция — это тип контента.

Примеры коллекций:

* `pages`
* `posts`
* `services`
* `faq`
* `contacts`
* `team`
* `products`

У коллекции есть:

* название;
* slug;
* описание;
* тип;
* порядок;
* поля;
* записи.

### Типы коллекций

#### `multiple`

Обычная коллекция с множеством записей.

Подходит для:

* новостей;
* статей;
* услуг;
* товаров;
* команды;
* FAQ.

Пример API:

```text
/?api=entries&c=posts
```

Ответ вернёт массив записей.

#### `single`

Коллекция только с одной записью.

Подходит для:

* главной страницы;
* настроек сайта;
* контактов;
* SEO-настроек;
* блока hero.

Пример API:

```text
/?api=entries&c=home
```

Ответ вернёт один объект или `null`.

### Важное ограничение

Тип коллекции нельзя менять после создания.

Это сделано правильно: если коллекция уже используется, смена `multiple` на `single` может сломать API фронтенда.

## Поля

Поля определяют структуру записи внутри коллекции.

Каждое поле имеет:

* метку;
* ключ;
* тип;
* обязательность;
* порядок;
* дополнительные настройки для `relation`.

### Типы полей

| Тип        | Для чего                  |
| ---------- | ------------------------- |
| `text`     | Короткий текст            |
| `textarea` | Длинный текст             |
| `html`     | HTML-контент              |
| `number`   | Число                     |
| `date`     | Дата                      |
| `bool`     | Да/нет                    |
| `url`      | Ссылка                    |
| `image`    | Изображение               |
| `file`     | Файл                      |
| `json`     | Произвольный JSON         |
| `relation` | Связь с другой коллекцией |

### Ключ поля

Ключ поля используется в API.

Например поле:

```text
Title
```

может иметь ключ:

```text
title
```

В JSON это будет:

```json
{
  "data": {
    "title": "О компании"
  }
}
```

### Автоматический slug ключа

CMS преобразует ключ в безопасный формат:

* приводит к нижнему регистру;
* транслитерирует кириллицу;
* заменяет лишние символы;
* дефисы в ключах полей заменяются на подчёркивания.

Например:

```text
Meta Description
```

станет:

```text
meta_description
```

## Записи

Запись — это конкретный контент внутри коллекции.

Например, в коллекции `pages` могут быть записи:

* `home`
* `about`
* `contacts`

У записи есть:

* заголовок;
* slug;
* статус;
* данные полей;
* дата создания;
* дата обновления.

### Статусы

#### `draft`

Черновик.

В публичном API не отдаётся.

#### `published`

Опубликовано.

Отдаётся через публичный API.

## Связи между коллекциями

Связь создаётся через поле типа `relation`.

Например:

* коллекция `pages`;
* поле `author`;
* целевая коллекция `team`;
* режим связи `single`.

Или:

* коллекция `posts`;
* поле `related_posts`;
* целевая коллекция `posts`;
* режим связи `multiple`.

### Настройки relation-поля

У relation-поля есть:

| Настройка         | Описание                                     |
| ----------------- | -------------------------------------------- |
| target collection | Коллекция, на записи которой можно ссылаться |
| relation mode     | Одна запись или несколько записей            |

### Режимы связи

#### `single`

Хранится один ID записи.

```json
{
  "author": 3
}
```

#### `multiple`

Хранится массив ID записей.

```json
{
  "related_posts": [2, 5, 8]
}
```

### Проверка связей

CMS проверяет:

1. Для поля `relation` обязательно должна быть выбрана целевая коллекция.
2. Связанная запись должна принадлежать выбранной целевой коллекции.
3. Нельзя сохранить ID записи из другой коллекции.
4. В админке рядом со связанной записью показывается её статус: `draft` или `published`.

### Раскрытие связей через `populate`

По умолчанию relation-поле отдаёт ID записи.

```text
/?api=entry&c=posts&s=my-post
```

Чтобы получить объект связанной записи, добавьте:

```text
populate=1
```

```text
/?api=entry&c=posts&s=my-post&populate=1
```

При `populate=1` публичный API раскрывает только опубликованные связанные записи.

## Файлы и изображения

CMS поддерживает загрузку файлов через поля:

* `image`
* `file`

### Ограничение размера

Максимальный размер файла:

```text
10 MB
```

### Разрешённые файлы

Для поля `file` разрешены:

```text
jpg, jpeg, png, webp, gif, pdf, doc, docx, xls, xlsx, ppt, pptx, txt, csv, zip, rar
```

Для поля `image` разрешены:

```text
jpg, jpeg, png, webp, gif
```

Для изображений дополнительно выполняется проверка, что файл действительно является изображением.

### Где хранятся файлы

Файлы сохраняются в папку:

```text
uploads/
```

Информация о файле хранится в таблице:

```text
files
```

### Очистка файлов

В разделе `Файлы` можно очистить неиспользуемые файлы.

CMS проверяет:

* какие файлы используются в записях;
* какие файлы лежат в `uploads/`;
* какие файлы есть в таблице `files`.

Неиспользуемые файлы удаляются из папки, а записи в базе помечаются как `deleted`.

* `tr`
* `de`

* `it`
* `pt`
* `zh`

В разделе:


* включить или выключить интернационализацию;

Через параметр:

```text

```

```text
/?api=entry&c=pages&s=home&lang=ru
```


```text
/?api=entry&c=pages&s=home&locale=kk
```

Если добавить:

```text
populate=1
```

то связи раскрываются не только в `data`, но и внутри `translations`.

## API

API работает через query-параметр:

```text
?api=...
```

Базовый URL зависит от того, где лежит файл CMS.

Например:

```text
https://example.com/?api=index
```

или локально:

```text
http://localhost:8000/?api=index
```

### Общие параметры API

| Параметр   | Описание                                   |
| ---------- | ------------------------------------------ |
| `api`      | Название API-метода                        |
| `c`        | Slug коллекции                             |
| `s`        | Slug записи                                |
| `g`        | Slug группы                                |
| `lang`     | Язык контента                              |
| `locale`   | Альтернатива `lang`                        |
| `populate` | Раскрывать связи: `1`, `true`, `yes`, `on` |
| `fields`   | Добавить поля/схему, где поддерживается    |
| `project`  | Slug проекта                               |
| `p`        | Короткая альтернатива `project`            |

### Публичные методы

Доступны без авторизации:

* `index`
* `entries`
* `entry`
* `group`

Публичные методы отдают только опубликованный контент.

### Приватные методы

Требуют авторизации и подходящей роли:

* `collections`
* `groups`
* `fields`
* `schema`
* `files`
* `files-trash`

## API: group

```text
GET /?api=group&g=main
```

С языком:

```text
GET /?api=group&g=main&lang=ru
```

С раскрытием связей:

```text
GET /?api=group&g=main&lang=ru&populate=1
```

В ответе группы есть два формата:

* `collections` — массив коллекций;
* `by_slug` — объект, где ключом является slug коллекции.

Во фронтенде удобно обращаться так:

```js
const pages = data.group.by_slug.pages.data;
```

## Примеры использования API

### Получить главную страницу

```js
async function getHome() {
  const res = await fetch('/?api=entry&c=pages&s=home&lang=ru&populate=1');
  const json = await res.json();

  if (!json.ok) {
    throw new Error(json.error || 'API error');
  }

  return json.data;
}
```

### Получить группу главной страницы

```js
async function getMainGroup() {
  const res = await fetch('/?api=group&g=main&lang=ru&populate=1');
  const json = await res.json();

  if (!json.ok) {
    throw new Error(json.error || 'API error');
  }

  return json.group.by_slug;
}
```

### Использование в обычном HTML

```html
<div id="app"></div>

<script>
fetch('/?api=entry&c=pages&s=home&lang=ru')
  .then(response => response.json())
  .then(json => {
    if (!json.ok) return;

    document.getElementById('app').innerHTML = json.data.data.content;
  });
</script>
```

## Ошибки API

### `setup_required`

CMS ещё не настроена.

```json
{
  "ok": false,
  "error": "setup_required",
  "message": "Выбор базы данных"
}
```

### `first_user_required`

База выбрана, но первый пользователь ещё не создан.

```json
{
  "ok": false,
  "error": "first_user_required",
  "message": "Первый пользователь"
}
```

### `auth_required`

Нет доступа к приватному API.

```json
{
  "ok": false,
  "error": "auth_required",
  "message": "Этот API доступен только авторизованным пользователям"
}
```

### `collection_not_found`

Коллекция не найдена.

```json
{
  "ok": false,
  "error": "collection_not_found"
}
```

### `entry_not_found`

Запись не найдена или не опубликована.

```json
{
  "ok": false,
  "error": "entry_not_found"
}
```

### `group_not_found`

Группа не найдена.

```json
{
  "ok": false,
  "error": "group_not_found"
}
```

### `api_not_found`

API-метод не найден.

```json
{
  "ok": false,
  "error": "api_not_found"
}
```

## Безопасность

В CMS уже есть базовые меры безопасности.

### Сессии

Используются настройки:

* `session.use_strict_mode`
* `session.cookie_httponly`
* `SameSite=Lax`
* `secure` cookie при HTTPS

### CSRF

Все POST-формы используют CSRF-токен.

Если токен неверный, запрос завершается ошибкой:

```text
CSRF
```

### Пароли

Пароли хранятся через:

```php
password_hash()
```

Проверка выполняется через:

```php
password_verify()
```

Минимальная длина пароля:

```text
10 символов
```

Максимальная длина:

```text
72 символа
```

### Ограничение входа

После нескольких неудачных попыток вход временно блокируется.

По умолчанию:

```text
5 неудачных попыток
10 минут блокировки
```

### Публичный API

Публичный API отдаёт только записи со статусом:

```text
published
```

Черновики не видны во внешнем API.

## Развёртывание

### Apache

Пример структуры:

```text
/var/www/cms/
├── index.php
├── storage/
└── uploads/
```

Права:

```bash
chown -R www-data:www-data /var/www/cms/storage /var/www/cms/uploads
chmod -R 775 /var/www/cms/storage /var/www/cms/uploads
```

### Nginx

Пример location:

```nginx
server {
    server_name example.com;
    root /var/www/cms;
    index index.php;

    location / {
        try_files $uri $uri/ /index.php?$query_string;
    }

    location ~ \.php$ {
        include snippets/fastcgi-php.conf;
        fastcgi_pass unix:/run/php/php8.3-fpm.sock;
    }
}
```

### Закрыть `storage` от браузера

Для Nginx:

```nginx
location /storage/ {
    deny all;
}
```

Для Apache можно добавить `.htaccess` в `storage/`:

```apache
Deny from all
```

## Частые проблемы

### Белый экран или DB error

Проверьте:

* включён ли `pdo`;
* включён ли `pdo_sqlite` или `pdo_mysql`;
* существует ли база MySQL;
* правильные ли логин и пароль MySQL;
* есть ли права на папку `storage`.

### Не загружаются файлы

Проверьте:

* права на папку `uploads`;
* лимит `upload_max_filesize` в PHP;
* лимит `post_max_size` в PHP;
* расширение файла;
* размер файла.

### API не отдаёт запись

Проверьте:

* запись существует;
* slug указан правильно;
* статус записи `published`;
* коллекция указана через правильный `c`;
* проект указан правильно, если используется `project`.

### Relation не сохраняется

Проверьте:

* у поля `relation` выбрана целевая коллекция;
* выбранная запись принадлежит этой целевой коллекции;
* запись не была удалена;
* режим связи выбран правильно: `single` или `multiple`.

### `populate=1` не раскрывает связь

Проверьте:

* поле действительно имеет тип `relation`;
* связанная запись существует;
* связанная запись опубликована;
* ID хранится в правильном формате;
* для multiple-связи хранится массив ID.

## Рекомендуемый сценарий использования

### Для простого сайта

1. Создайте проект `default`.
2. Создайте группу `main`.
3. Создайте коллекции:

   * `pages`
   * `services`
   * `faq`
   * `contacts`
4. Добавьте коллекции в группу `main`.
5. Во фронтенде используйте:

```text
/?api=group&g=main&lang=ru&populate=1
```

### Для главной страницы

Можно создать `single` коллекции:

* `home_hero`
* `home_about`
* `home_contacts`

И добавить их в группу:

```text
main
```

Так фронтенд будет получать всю главную страницу одним запросом.

### Для блога

Создайте:

* `posts`
* `categories`
* `authors`

В `posts` добавьте relation-поля:

* `category` → `categories`, single
* `author` → `authors`, single
* `related_posts` → `posts`, multiple

После этого можно получать пост так:

```text
/?api=entry&c=posts&s=my-post&lang=ru&populate=1
```

## Коротко

Mini Headless CMS — это лёгкая PHP CMS для управления структурированным контентом и выдачи данных через JSON API.

Она подходит, если нужно:

* быстро подключить админку к сайту;
* не ставить тяжёлую CMS;
* хранить контент в SQLite или MySQL;
* отдавать данные во фронтенд через API;
* управлять мультиязычным контентом;
* связывать коллекции между собой;
* получать несколько коллекций одним запросом через группы.

Главная идея проекта: простая, быстрая и понятная headless CMS без лишней сложности.
