## О проекте

**Mini Headless CMS** — компактная CMS для разработчиков, фрилансеров и небольших проектов, которым нужна простая админ-панель и JSON API без установки WordPress, Strapi, Directus или отдельного backend-приложения.

Основная идея проекта:

> загрузить один PHP-файл на обычный хостинг, пройти первоначальную настройку и сразу получить админ-панель, базу данных, управление контентом, формами и публичный API.

CMS не привязана к конкретному frontend-стеку. Её можно использовать с обычным HTML, JavaScript, Vue, Nuxt, React, Next.js и любым другим клиентом, который умеет отправлять HTTP-запросы.

### Когда проект особенно полезен

- сайт размещён на обычном PHP-хостинге;
- клиенту нужна понятная админ-панель;
- frontend уже написан отдельно;
- полноценная CMS слишком тяжёлая или дорогая;
- нужно быстро добавить редактируемый контент;
- нужно принимать заявки с сайта без отдельного backend;
- не хочется поднимать VPS, Docker, Node.js и дополнительные сервисы.

### Чем Mini Headless CMS не является

Это не визуальный конструктор сайтов и не замена полноценной enterprise-CMS. Проект ориентирован на небольшие и средние сайты, лендинги, каталоги, корпоративные страницы и кастомные frontend-приложения.

## Архитектура

Основная логика CMS находится в одном PHP-файле:

```text
index.php
```

Внутри него находятся:

- первичная установка;
- подключение к базе данных;
- создание и обновление таблиц;
- авторизация;
- роли и разрешения;
- маршрутизация админ-панели;
- CRUD-операции;
- публичный JSON API;
- обработчик форм;
- загрузка файлов;
- HTML, CSS и JavaScript интерфейса.

Рабочие данные создаются отдельно:

```text
mini-cms/
├── index.php
├── storage/
│   ├── config.json
│   ├── cms.sqlite
│   └── login_attempts.json
└── uploads/
```

Такой подход позволяет быстро перенести CMS на другой хостинг: достаточно скопировать основной файл, `storage/` и `uploads/`.

## Требования

- PHP `8.0` или новее;
- PDO;
- `pdo_sqlite` для SQLite;
- `pdo_mysql` для MySQL;
- `mbstring`;
- `json`;
- `fileinfo` рекомендуется;
- `dom` рекомендуется для расширенной очистки HTML;
- веб-сервер Apache, Nginx или встроенный PHP-сервер;
- права на запись в `storage/` и `uploads/`.

Интерфейс использует Bootstrap 5.3.3 и Bootstrap Icons 1.11.3 через jsDelivr CDN. Для полностью автономной установки замените CDN-ссылки локальными файлами.

## Быстрый старт

### Вариант 1. Клонирование репозитория

```bash
git clone https://github.com/DreamerView/mini-cms.git
cd mini-cms
```

Создайте рабочие каталоги:

```bash
mkdir -p storage uploads
chmod -R 775 storage uploads
```

Запустите локальный сервер:

```bash
php -S localhost:8000
```

Откройте:

```text
http://localhost:8000
```

### Вариант 2. Обычный shared-хостинг

1. Загрузите основной PHP-файл в отдельную папку сайта.
2. Назовите его `index.php`.
3. Создайте каталоги `storage` и `uploads`.
4. Разрешите PHP запись в эти каталоги.
5. Откройте CMS в браузере.
6. Выберите SQLite или MySQL.
7. Создайте первого администратора.

Пример:

```text
public_html/cms/
├── index.php
├── storage/
└── uploads/
```

CMS будет доступна по адресу:

```text
https://example.com/cms/
```

## Первый запуск

Первичная настройка проходит в два шага.

### 1. Выбор базы данных

#### SQLite

Рекомендуемый вариант для быстрого старта и небольших проектов.

CMS автоматически создаст:

```text
storage/cms.sqlite
```

#### MySQL

Подходит, когда база данных уже предоставляется хостингом или требуется отдельное серверное хранилище.

Базу необходимо создать заранее. Таблицы CMS создаст самостоятельно.

После выбора драйвера настройки сохраняются в:

```text
storage/config.json
```

### 2. Создание администратора

Если таблица пользователей пустая, CMS предложит создать первого пользователя.

Первый пользователь получает роль:

```text
admin
```

Требования к паролю:

- от 10 до 72 символов;
- разрешены буквы, цифры и специальные символы.

## Важная настройка безопасности

Каталог `storage/` содержит конфигурацию и базу SQLite. Он не должен быть доступен напрямую через браузер.

### Apache

Создайте файл:

```text
storage/.htaccess
```

Содержимое:

```apache
Require all denied
```

Дополнительно можно запретить выполнение PHP в `uploads/`:

```apache
<FilesMatch "\.(php|phtml|phar)$">
    Require all denied
</FilesMatch>
```

### Nginx

Добавьте в конфигурацию сайта:

```nginx
location ^~ /storage/ {
    deny all;
    return 404;
}

location ~* ^/uploads/.*\.(php|phtml|phar)$ {
    deny all;
    return 404;
}
```

Для максимальной защиты можно изменить константы путей и вынести `storage/` за пределы публичного каталога сайта.

## Основные сущности

### Проект

Проект — отдельное рабочее пространство внутри общей базы.

Каждый проект имеет собственные:

- коллекции;
- разделы;
- записи;
- формы;
- заявки;
- файлы.

Это позволяет использовать одну установку CMS для нескольких небольших сайтов или отдельных частей одного продукта.

### Коллекция

Коллекция описывает источник структурированных данных.

Примеры:

```text
pages
posts
products
services
team
faq
settings
```

Коллекция существует независимо от разделов и может использоваться повторно.

#### `multiple`

Содержит несколько записей.

Подходит для:

- статей;
- услуг;
- товаров;
- сотрудников;
- FAQ;
- новостей.

#### `single`

Содержит только одну запись.

Подходит для:

- главной страницы;
- контактов;
- настроек сайта;
- SEO-настроек;
- hero-блока.

Тип коллекции фиксируется после создания, чтобы случайно не изменить контракт API.

### Раздел контента

Раздел — это подборка или ярлык для коллекций.

Например:

```text
Главная страница
Каталог
Блог
Системные данные
```

Одна коллекция может быть добавлена сразу в несколько разделов. Удаление связи с разделом не удаляет коллекцию, поля, записи или файлы.

### Поле

Поле определяет структуру данных записи.

Поддерживаемые типы:

| Тип | Назначение |
|---|---|
| `text` | Короткий текст |
| `textarea` | Длинный текст |
| `html` | HTML-контент с preview |
| `number` | Число |
| `date` | Дата |
| `bool` | Логическое значение |
| `url` | Ссылка |
| `image` | Изображение |
| `file` | Документ или архив |
| `json` | Произвольный JSON |
| `relation` | Связь с другой коллекцией |

Для поля можно задать:

- название;
- ключ;
- тип;
- обязательность;
- порядок;
- целевую коллекцию и режим для `relation`.

Ключ, тип и параметры связи блокируются после создания поля. Это защищает уже сохранённые данные от несовместимого изменения схемы.

### Запись

Запись содержит:

- заголовок;
- slug;
- статус;
- значения полей;
- автора последнего изменения;
- дату создания;
- дату обновления.

Статусы:

- `draft` — виден в админке, но не возвращается публичным API;
- `published` — доступен публичному API.

## Пресеты коллекций

При создании коллекции можно использовать готовую схему:

| Пресет | Пример полей |
|---|---|
| `Blank` | Пустая коллекция |
| `Page` | HTML-контент, meta description |
| `Blog` | Обложка, описание, контент, дата публикации |
| `Product` | Изображение, цена, описание, наличие |
| `FAQ` | Вопрос, ответ |
| `Contacts` | Телефон, email, адрес, ссылка на карту |

Пресет только создаёт начальный набор полей. После этого схему можно расширять самостоятельно.

---

## Связи между коллекциями

Поле `relation` связывает запись с одной или несколькими записями другой коллекции.

Режимы:

```text
single
multiple
```

Пример без раскрытия связи:

```json
{
  "author": 12,
  "categories": [3, 7]
}
```

Добавьте параметр:

```text
populate=1
```

и API вернёт связанные опубликованные записи объектами.

```text
GET /?api=entry&project=default&c=posts&s=hello-world&populate=1
```

CMS проверяет, что выбранные записи действительно принадлежат целевой коллекции.

## Мультиязычный контент

Языки интерфейса:

- русский;
- қазақша;
- English.

Поддерживаемые языки контента:

```text
ru, kk, en, tr, de, fr, es, it, pt, zh, ja, ko, ar, hi, uz
```

В настройках можно:

- включить или выключить интернационализацию;
- выбрать активные языки;
- выбрать язык по умолчанию.

Получение конкретного языка:

```text
GET /?api=entry&project=default&c=pages&s=home&lang=ru
```

Альтернативный параметр:

```text
locale=kk
```

Если язык не передан, используется язык контента по умолчанию.

Отключение языка не удаляет его данные из базы. Они остаются сохранёнными, но скрываются из редактора.

## Автосохранение и история

Редактор записей поддерживает:

- фоновое автосохранение текста;
- восстановление незавершённого черновика;
- предупреждение о несохранённых изменениях;
- JSON-preview;
- историю изменений;
- список изменённых полей;
- автора изменения;
- восстановление старой версии.

Файлы не входят в автосохранение и загружаются только после нажатия основной кнопки **Сохранить**.

Автосохранённые черновики старше 30 дней удаляются обслуживанием данных.

## Файлы

Максимальный размер одного файла по умолчанию:

```text
10 MB
```

Разрешённые расширения:

```text
jpg, jpeg, png, webp, gif,
pdf, doc, docx, xls, xlsx,
ppt, pptx, txt, csv, zip, rar
```

Для поля `image` разрешены:

```text
jpg, jpeg, png, webp, gif
```

CMS дополнительно проверяет, что загруженный файл действительно является изображением.

### Корзина

Неиспользуемые файлы сначала перемещаются в корзину и физически остаются на диске.

Из корзины файл можно:

- восстановить;
- удалить окончательно.

После удаления проекта связанные с ним файлы перемещаются в глобальную корзину, а не исчезают безвозвратно.

## Формы и заявки

Формы позволяют принимать данные с любого сайта без отдельного обработчика.

Пример публичного endpoint:

```text
https://example.com/cms/index.php?form=contact&project=default
```

### Получение схемы формы

```http
GET /cms/index.php?form=contact&project=default
```

Пример ответа:

```json
{
  "ok": true,
  "form": {
    "name": "Обратная связь",
    "slug": "contact",
    "description": "Форма сайта",
    "fields": [
      {
        "label": "Имя",
        "key": "name",
        "type": "text",
        "required": true
      },
      {
        "label": "Email",
        "key": "email",
        "type": "email",
        "required": true
      }
    ]
  },
  "method": "POST",
  "files": false
}
```

### Обычная HTML-форма

```html
<form
  action="https://example.com/cms/index.php?form=contact&project=default"
  method="post"
>
  <input type="text" name="name" required>
  <input type="email" name="email" required>
  <textarea name="message"></textarea>

  <input
    type="text"
    name="_website"
    tabindex="-1"
    autocomplete="off"
    hidden
  >

  <button type="submit">Отправить</button>
</form>
```

### Отправка JSON

```js
const response = await fetch(
  'https://example.com/cms/index.php?form=contact&project=default',
  {
    method: 'POST',
    headers: {
      'Content-Type': 'application/json'
    },
    body: JSON.stringify({
      name: 'Темирхан',
      email: 'hello@example.com',
      message: 'Хочу обсудить проект'
    })
  }
);

const result = await response.json();
```

### Поддерживаемые типы полей формы

```text
text
textarea
email
tel
number
integer
boolean
date
datetime
url
json
```

### Ограничения обработчика форм

- файлы пока не принимаются;
- максимальный JSON-размер данных — около 64 KB;
- не более 20 отправок за 10 минут с одного IP для одной формы;
- лишние поля не сохраняются, если у формы задана схема;
- заполненный `_hp` или `_website` помечает заявку как spam;
- публичный endpoint разрешает CORS-запросы.

## JSON API

API работает через query-параметр:

```text
?api=...
```

Базовый пример:

```text
https://example.com/cms/index.php?api=index&project=default
```

### Публичные маршруты

| Маршрут | Назначение |
|---|---|
| `api=index` | Информация о CMS и доступных маршрутах |
| `api=entries` | Опубликованные записи коллекции |
| `api=entry` | Одна опубликованная запись |
| `api=group` | Несколько коллекций одного раздела |

### Приватные маршруты

Приватные методы используют авторизованную сессию CMS.

| Маршрут | Назначение |
|---|---|
| `api=collections` | Список коллекций |
| `api=groups` | Список разделов |
| `api=fields` | Поля коллекции |
| `api=schema` | Полная схема коллекции |
| `api=files` | Активные файлы |
| `api=files-trash` | Корзина файлов |

### Общие параметры

| Параметр | Описание |
|---|---|
| `project` | Slug проекта |
| `p` | Короткий вариант `project` |
| `c` | Slug коллекции |
| `s` | Slug записи |
| `g` | Slug раздела |
| `lang` | Язык контента |
| `locale` | Альтернатива `lang` |
| `populate=1` | Раскрыть relation-поля |
| `fields=1` | Добавить схему, требует доступа |

### Получить записи коллекции

```http
GET /cms/index.php?api=entries&project=default&c=posts&lang=ru
```

### Получить одну запись

```http
GET /cms/index.php?api=entry&project=default&c=pages&s=home&lang=ru
```

### Получить раздел целиком

```http
GET /cms/index.php?api=group&project=default&g=main&lang=ru&populate=1
```

Пример JavaScript:

```js
async function getMainContent() {
  const response = await fetch(
    '/cms/index.php?api=group&project=default&g=main&lang=ru&populate=1'
  );

  const json = await response.json();

  if (!json.ok) {
    throw new Error(json.error || 'CMS API error');
  }

  return json.group.by_slug;
}
```

## Почему разделы полезны для API

Допустим, главная страница состоит из коллекций:

```text
hero
advantages
services
faq
contacts
```

Вместо пяти запросов можно добавить эти коллекции в раздел `main` и получить их одним запросом:

```text
GET /?api=group&g=main
```

Ответ содержит:

- `collections` — упорядоченный массив;
- `by_slug` — объект для быстрого доступа по slug.

Пример:

```js
const content = response.group.by_slug;

const hero = content.hero.data;
const services = content.services.data;
const faq = content.faq.data;
```

### Роли

- **Admin** — полный доступ.
- **Developer** — схема, связи, проекты, формы и API; записи доступны только для просмотра.
- **Editor** — контент, разделы, файлы и заявки без изменения схемы.
- **Viewer** — только просмотр.

Текущего пользователя нельзя удалить или отключить.

## Защита

В текущей версии реализованы:

- `HttpOnly` session cookie;
- `SameSite=Lax`;
- `Secure` cookie при HTTPS;
- строгий режим сессий;
- CSRF-токены для действий админки;
- хеширование паролей через `password_hash`;
- блокировка после 5 неудачных попыток входа;
- блокировка входа на 10 минут;
- подготовленные PDO-запросы;
- разграничение прав;
- фильтрация загружаемых файлов;
- проверка изображений;
- очистка HTML;
- ограничение размера данных форм;
- rate limit форм;
- honeypot против простых ботов;
- публичная выдача только опубликованного контента.

### Рекомендации для production

- обязательно используйте HTTPS;
- закройте `storage/` от прямого доступа;
- создавайте резервные копии базы и файлов;
- не храните MySQL-пароль в публично доступном каталоге;
- ограничьте доступ к админке на уровне сервера, если это возможно;
- проверьте значения `upload_max_filesize` и `post_max_size`;
- замените CDN-ресурсы локальными, если нужна автономная работа;
- протестируйте права всех ролей перед передачей клиенту.

## Резервное копирование

### SQLite

Скопируйте:

```text
storage/cms.sqlite
storage/config.json
uploads/
```

Перед копированием желательно временно остановить запись данных или создать консистентную копию средствами SQLite.

### MySQL

Экспортируйте базу:

```bash
mysqldump -u USER -p DATABASE > backup.sql
```

И отдельно сохраните:

```text
storage/config.json
uploads/
```

## Сброс выбора базы

В настройках можно удалить текущий выбор базы.

При сбросе удаляется только:

```text
storage/config.json
```

Не удаляются:

- SQLite-база;
- загруженные файлы;
- содержимое MySQL.

После сброса CMS снова покажет экран первоначального выбора базы.

## Текущие ограничения

Проект находится в активной разработке.

На данный момент:

- API построен на query-параметрах, а не на REST-маршрутах;
- приватный API использует сессию админки, API-токенов пока нет;
- формы не принимают файлы;
- нет встроенной отправки email и webhook;
- нет автоматического планировщика резервных копий;
- нет отдельной системы миграций версий;
- интерфейс загружает Bootstrap через CDN;
- SQLite и MySQL — единственные официально поддерживаемые базы;
- архитектура одного файла удобна для установки, но усложняет командную разработку крупных расширений.

## Разработка
