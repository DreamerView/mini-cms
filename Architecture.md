# Mini Headless CMS — карта архитектуры и производительности

## Формат проекта

CMS остаётся одним переносимым PHP-файлом. Внутри он разделён на независимые слои, чтобы дальнейшие изменения не смешивали инфраструктуру, бизнес-логику и HTML.

## Основные модули

### `RequestCache`
Кэширует данные только в рамках текущего HTTP-запроса. Убирает повторные SQL-запросы к проектам, коллекциям, полям, записям, файлам и relation-связям.

### `ConfigStore`
Читает `storage/config.json` один раз за запрос и сохраняет его атомарно через временный файл и `rename`.

### `ResponseCache`
Файловый кэш публичного API. Поддерживает:

- ETag и ответ `304 Not Modified`;
- `Last-Modified`;
- `stale-while-revalidate`;
- версионную инвалидацию после изменения публичных данных.

### `Database`
Единая точка работы с PDO:

- одно соединение за запрос;
- подготовленные выражения;
- транзакции;
- учёт числа и времени SQL-запросов;
- очистка request-cache после записи;
- инвалидация публичного API-кэша только при изменении контента.

### `Runtime`
Настраивает PHP и запускает сессию только там, где она действительно нужна. Публичные GET-запросы API и форм не блокируются сессионным файлом.

### `AppKernel`
Центральный маршрутизатор приложения. Отвечает за последовательность: настройка → база → обслуживание → API → авторизация → административные страницы.

## Оптимизация базы данных

- Миграции выполняются только при изменении `APP_SCHEMA_VERSION`.
- Добавлены индексы для часто используемых фильтров, сортировок и связей.
- Списки используют агрегированные JOIN-запросы вместо коррелированных подзапросов.
- Публичный API применяет SQL-пагинацию `COUNT + LIMIT/OFFSET` там, где не требуется поиск внутри JSON.
- Relation-поля загружаются пакетно по коллекциям, а не отдельным запросом для каждого ID.
- Проверка нескольких relation-ID выполняется одним запросом `IN (...)`.
- Связанные файлы также предварительно загружаются пакетно.

## Оптимизация TTFB и API

- Публичные маршруты не запускают PHP-сессию без необходимости.
- После чтения авторизации GET-запрос API закрывает session lock.
- Конфигурация и наиболее часто запрашиваемые сущности кэшируются в памяти запроса.
- Публичные JSON-ответы кэшируются на диске.
- JSON в production выдаётся без лишнего форматирования.
- Изменение журнала, черновика или попыток входа не сбрасывает кэш публичного API.

## Оптимизация LCP и Lighthouse

- Большие CSS и JavaScript вынесены во внутренние endpoint-ресурсы этого же PHP-файла.
- Ресурсы имеют версию, ETag и `Cache-Control: immutable` на один год.
- Bootstrap Icons загружается неблокирующим способом.
- Для CDN добавлены `preconnect` и `dns-prefetch`.
- Тема устанавливается сразу на корневом `<html>`, что уменьшает визуальное мигание.
- Стартовая HTML-страница в локальном тесте уменьшилась примерно со 137 КБ до 23 КБ; CSS и JS после первого посещения берутся из браузерного кэша.

## Инвалидация кэша

При изменении проектов, коллекций, полей, записей, разделов, форм или файлов обновляется `storage/cache/revision`. Старые API-файлы перестают использоваться автоматически. Ручное удаление каталога `storage/cache` безопасно: CMS создаст его снова.

## Рекомендуемая конфигурация production

- PHP 8.2 или новее;
- включённый OPcache;
- `display_errors=Off`, `log_errors=On`;
- Brotli или gzip на веб-сервере;
- HTTP/2 или HTTP/3;
- HTTPS;
- для SQLite — локальный SSD и недоступный из Web каталог `storage`;
- для высокой параллельной нагрузки — MySQL;
- CDN или локальная раздача Bootstrap при необходимости полностью исключить внешний DNS-запрос.

## Правила дальнейшего развития

1. Инфраструктурный код добавлять в классы верхней части файла.
2. Чтение сущностей проводить через существующие repository-функции, чтобы работал request-cache.
3. Все изменения данных выполнять через `q()`/`run()` — тогда кэш инвалидируется автоматически.
4. Публичные GET-ответы отдавать через `J_api()`.
5. При изменении CSS или JavaScript повышать `APP_CACHE_VERSION`.
6. При изменении структуры таблиц повышать `APP_SCHEMA_VERSION`.
7. Не выполнять SQL внутри циклов, если данные можно получить одним `JOIN` или `IN (...)`.
